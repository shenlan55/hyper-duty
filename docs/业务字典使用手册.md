# Hyper Duty 业务字典使用手册

> 适用范围：Hyper Duty 全平台业务枚举的统一管理规范
> 文档位置：`docs/业务字典使用手册.md`
> 维护规范：每次新增业务枚举，必须同步更新本文档"已注册字典清单"

---

## 1. 设计原则

**业务枚举（任务状态/班次/审批/是否/启用禁用/性别 等）必须统一走 sys_dict_type + sys_dict_data，**禁止在前端 Vue 组件、后端 Java 常量中硬编码 label/value 映射。

| 维度 | 旧做法（禁止） | 新做法（统一） |
|------|--------------|--------------|
| 业务 label | `<el-tag>{{ STATUS_MAP[row.status] }}</el-tag>` 写死 Map | `<el-tag>{{ statusLabel(row.status) }}</el-tag>` 走 useDict |
| 业务 value | `if (status === 1) { ... }` 写死 if/else | 后端按 `dict_value` 字段存储，前端 `useDict('task_status')` |
| 新增枚举项 | 改前端 .vue + 改后端 Java enum + 改 SQL | **只在 sys_dict_data 加一行 + 刷新缓存** |
| 下拉选项来源 | 组件内写死 `<el-option v-for="...">` | 组件内 `useDict(...).options` |

### 1.1 必须走字典的枚举类型

凡满足以下任一条件，都必须走字典：

1. **多业务模块共用**（如"是否"、"启用/禁用"、性别、审批状态）
2. **会频繁增减/调整**（如请假类型、任务优先级）
3. **需要支持后台可视化维护**（管理员可在「系统管理→字典管理」页面增删）
4. **前后端 label 必须保持一致**（如"白班/早班/中班/晚班"）

### 1.2 暂不走字典的枚举

- 状态机内部强约束（如工作流实例的 `active/suspended/finished`，只用于引擎内部判断）
- 纯数字标记位（如 `is_deleted`、`has_children` 这种 0/1 业务位）
- 一次性内部枚举（如某个导出格式的 1/2/3）

## 2. 数据库设计

### 2.1 表结构

```sql
-- 字典类型表（业务枚举分类）
CREATE TABLE IF NOT EXISTS public.sys_dict_type (
    id          BIGSERIAL PRIMARY KEY,
    dict_name   VARCHAR(100) NOT NULL,  -- 中文名（如"班次类型"）
    dict_code   VARCHAR(100) NOT NULL,  -- 英文编码（前端 useDict 唯一标识）
    description VARCHAR(500),           -- 说明
    status      SMALLINT DEFAULT 1,     -- 1=启用 0=禁用
    create_time TIMESTAMP,
    update_time TIMESTAMP,
    CONSTRAINT uk_dict_code UNIQUE (dict_code)
);

-- 字典数据表（具体枚举项）
CREATE TABLE IF NOT EXISTS public.sys_dict_data (
    id            BIGSERIAL PRIMARY KEY,
    dict_type_id  BIGINT NOT NULL,        -- 关联 sys_dict_type.id
    dict_label    VARCHAR(100) NOT NULL,  -- 显示文本（如"白班"）
    dict_value    VARCHAR(100) NOT NULL,  -- 业务值（如"0"）
    dict_sort     INT DEFAULT 0,          -- 排序（升序）
    css_class     VARCHAR(50),            -- 预留 CSS
    list_class    VARCHAR(50),            -- el-tag 类型：primary/success/warning/danger/info
    is_default    SMALLINT DEFAULT 0,     -- 是否默认项（1=是）
    status        SMALLINT DEFAULT 1,     -- 1=启用 0=禁用
    remark        VARCHAR(500),
    create_time   TIMESTAMP,
    update_time   TIMESTAMP
);
```

### 2.2 命名规范

| 字段 | 规范 | 示例 |
|------|------|------|
| `dict_code` | snake_case，全局唯一 | `task_status`、`shift_type`、`approval_status` |
| `dict_label` | 中文，简洁 | "未开始"、"白班"、"事假" |
| `dict_value` | 业务侧需要的原始值，**统一存字符串** | "0"/"1"/"2" 或 "daily"/"weekly" |
| `list_class` | 5 选 1 | `primary` / `success` / `warning` / `danger` / `info` |

> 关键：**`dict_value` 统一存字符串**。前端 `getLabel` 内部 `String(value)` 兼容，业务表 `status` 字段可以保持 `INTEGER` 存储 `0/1/2`，只在比较时统一转字符串。

## 3. 后端 API

### 3.1 批量按编码查询

```
GET /api/dict/data/byCodes?codes=task_status,task_priority
```

**响应**（响应拦截器已 unwrap）：
```json
{
  "task_status": [
    { "id": 110, "dictTypeId": 11, "dictLabel": "未开始", "dictValue": "0", "listClass": "info", ... },
    { "id": 111, "dictTypeId": 11, "dictLabel": "进行中", "dictValue": "1", "listClass": "primary", ... }
  ],
  "task_priority": [
    { "id": 120, "dictTypeId": 12, "dictLabel": "高", "dictValue": "1", "listClass": "danger", ... },
    ...
  ]
}
```

### 3.2 单编码查询

```
GET /api/dict/data/byCode/{code}
```

### 3.3 字典类型/数据 CRUD（系统管理→字典管理）

- `GET /api/dict/type/page` - 分页查字典类型
- `POST /api/dict/type` - 新增字典类型
- `PUT /api/dict/type` - 修改字典类型
- `DELETE /api/dict/type/{id}` - 删除字典类型
- `GET /api/dict/data/page` - 分页查字典数据
- `POST /api/dict/data` - 新增字典数据
- `PUT /api/dict/data` - 修改字典数据
- `DELETE /api/dict/data/{id}` - 删除字典数据

### 3.4 缓存

字典数据带 Redis 缓存，key 格式：`dict::data_{dictTypeId}`。
**修改字典后需清空缓存**（参考 `SysDictDataServiceImpl` 中的缓存清理逻辑）。

## 4. 前端使用

### 4.1 三件套

| 角色 | 文件 | 职责 |
|------|------|------|
| Pinia store | `frontend/src/stores/dict.js` | 全局缓存 + 批量加载 |
| Composable | `frontend/src/composables/useDict.js` | 组件内 `useDict('task_status')` 拿到 options/labelOf/tagTypeOf/loadDict |
| API 封装 | `frontend/src/api/dict.js` | 调后端接口 |
| 常量 | `frontend/src/stores/dict.js` → `DICT_CODES` | 字典 code 统一常量（避免拼写错误） |

### 4.2 标准使用模式

```vue
<script setup>
import { onMounted } from 'vue'
import { useDict } from '@/composables/useDict'

// 单字典
const { options: statusOptions, labelOf: statusLabel, tagTypeOf: statusType, loadDict: loadStatusDict } = useDict('task_status')

// 多字典
const { loadDict: loadAllDict } = useDict(['task_status', 'task_priority', 'task_focus'])

onMounted(async () => {
  await loadStatusDict()
  // 或批量：
  // await loadAllDict()
})
</script>

<template>
  <!-- el-select 下拉 -->
  <el-select v-model="form.status">
    <el-option v-for="opt in statusOptions" :key="opt.value" :label="opt.label" :value="opt.value" />
  </el-select>

  <!-- el-tag 显示 -->
  <el-tag :type="statusType(row.status)">{{ statusLabel(row.status) }}</el-tag>

  <!-- 直接取值（无字典时返回 fallback 或原 value） -->
  <span>{{ statusLabel(row.status, '未知') }}</span>
</template>
```

### 4.3 字典常量

新增业务枚举后，必须在 `frontend/src/stores/dict.js` 的 `DICT_CODES` 中添加常量：

```javascript
export const DICT_CODES = Object.freeze({
  COMMON_STATUS: 'common_status',
  TASK_STATUS: 'task_status',
  TASK_PRIORITY: 'task_priority',
  TASK_FOCUS: 'task_focus',
  // ... 新增时在这里加一行
})
```

### 4.4 注意事项

- `loadDict` **不会**自动调用，需要在 `onMounted` 或 `watch` 中显式触发
- 同一个 dict_code **多次 loadDict 是安全的**（Pinia store 内部去重 + loadingSet 防并发）
- 未加载字典时 `getLabel(value)` 会返回 fallback 或原 value，**不会抛错**
- 字典加载失败时控制台会输出 `[dict] loadDict failed: ...`，需要排查后端 `/api/dict/data/byCodes` 是否可访问

## 5. 新增业务枚举的标准流程

### 5.1 第一步：数据库初始化

在 `src/main/resources/sql/hyper_duty_dml.sql`（或新增模块的 DML）中插入字典类型和数据：

```sql
INSERT INTO public.sys_dict_type (id, dict_name, dict_code, description, status, create_time, update_time) VALUES
(33, '审批结果', 'approval_result', '通用审批-审批结果', 1, '2026-06-10 00:00:00', '2026-06-10 00:00:00');

INSERT INTO public.sys_dict_data (id, dict_type_id, dict_label, dict_value, dict_sort, list_class, is_default, status, remark, create_time, update_time) VALUES
(330, 33, '同意', '1', 1, 'success', 0, 1, '', '2026-06-10 00:00:00', '2026-06-10 00:00:00'),
(331, 33, '拒绝', '2', 2, 'danger',  0, 1, '', '2026-06-10 00:00:00', '2026-06-10 00:00:00'),
(332, 33, '驳回', '3', 3, 'warning', 0, 1, '', '2026-06-10 00:00:00', '2026-06-10 00:00:00');
```

> 注意：id 必须全局唯一（推荐按模块分配段：系统 1-99、业务 100+）。执行 SQL 前确认 `uk_dict_code` 不冲突。

### 5.2 第二步：清除 Redis 缓存

修改字典数据后必须清缓存：

```bash
redis-cli DEL "dict::data_33"
```

或在 `SysDictDataServiceImpl` 中通过 service 层操作。

### 5.3 第三步：前端常量 + 组件接入

1. `frontend/src/stores/dict.js` 的 `DICT_CODES` 加一行：
   ```javascript
   APPROVAL_RESULT: 'approval_result',
   ```
2. 业务组件内：
   ```javascript
   const { options, labelOf, tagTypeOf, loadDict } = useDict('approval_result')
   onMounted(() => loadDict())
   ```

### 5.4 第四步：同步更新本文档

在"已注册字典清单"表格中添加新字典条目。

### 5.5 第五步：测试用例（至少 10 条）

- [ ] 后端 `/api/dict/data/byCodes?codes=approval_result` 返回正确数据
- [ ] 字典管理页面可正常增删改
- [ ] 业务页面下拉框渲染所有 option
- [ ] el-tag 显示 label 正确
- [ ] 业务表存储 `dict_value` 字符串后回显正确
- [ ] 修改字典 label 后刷新缓存生效
- [ ] 并发加载同一字典不重复请求
- [ ] 字典加载失败时页面不崩
- [ ] 业务表的 `dict_value` 字段与字典值一致
- [ ] 跨模块复用（如值班和项目都用 `yes_no`）正常

## 6. 初始化脚本使用

### 6.1 `hyper_duty_dict_init.sql`

**位置**：`src/main/resources/sql/hyper_duty_dict_init.sql`

**用途**：在已有数据库基础上快速写入/重置本平台全套业务字典（id=10~32 范围）。

**使用方式**：
1. 在 DBeaver 中打开该脚本
2. **整段全选执行**
3. 脚本会自动：
   - DELETE `id BETWEEN 10 AND 32` 范围（保留 id=1, 2, 5 等业务字典）
   - INSERT 20 个字典类型 + 60+ 条字典数据

**注意事项**：
- 脚本是**幂等**的：先 DELETE 再 INSERT，可重复执行
- 脚本**不依赖** `ON CONFLICT` 语法（兼容 DBeaver/部分代理工具）
- 业务表（如 `duty_shift_config`、`sys_employee`）的 `dict_value` 引用**不会**受影响，因为脚本只动 `sys_dict_*` 两张表
- 如果数据库中已有自定义字典（id > 32），**不会**被清除

### 6.2 与主 DML 的关系

| 文件 | 用途 |
|------|------|
| `hyper_duty_ddl.sql` | 全平台表结构（含 sys_dict_type / sys_dict_data 建表） |
| `hyper_duty_dml.sql` | 平台基础数据（含 id=1, 2, 5 等最小化字典） |
| `hyper_duty_dict_init.sql` | **本手册配套**，快速注入全套业务字典（id=10~32） |

**新装环境**：依次执行 `hyper_duty_ddl.sql` → `hyper_duty_dml.sql` → `hyper_duty_dict_init.sql`。

**已有环境升级**：单独执行 `hyper_duty_dict_init.sql` 即可。

## 7. 踩坑经验

### 7.1 字典 SQL 重复 key 错误

**症状**：`ERROR: duplicate key value violates unique constraint "uk_dict_code"`
**原因**：sys_dict_type 中 `dict_code` 唯一约束冲突
**修复**：
1. 用 `psql` 或 DBeaver 执行 `SELECT dict_code, COUNT(*) FROM sys_dict_type GROUP BY dict_code HAVING COUNT(*) > 1;` 找重复
2. 保留一个 id，删除另一个

### 7.2 字典 SQL 外键约束错误

**症状**：`ERROR: insert or update on table "sys_dict_data" violates foreign key constraint`
**原因**：sys_dict_data 的 `dict_type_id` 在 sys_dict_type 中不存在
**修复**：执行 INSERT 前先确认 dict_type_id 已存在，或调整 id 顺序

### 7.3 前端 useDict 后台调试后不生效

**原因**：Redis 缓存未清
**修复**：`redis-cli DEL "dict::data_{dictTypeId}"` 或在字典管理页面点"刷新"

### 7.4 onMounted 没调 loadDict 导致页面显示原 value

**症状**：页面显示 `0`、`1` 而不是"未开始"、"进行中"
**原因**：组件 setup 中 `useDict('xxx')` 只暴露了 `loadDict` 函数，需要显式触发
**修复**：在 `onMounted` 中 `await loadDict()`

### 7.5 BaseTable 翻页事件被吞

**症状**：点击翻页按钮不发请求，URL 不变
**原因**：`el-pagination` 用 `v-model:current-page` 时，`@current-change` 触发时本地 ref 已被 el-pagination 提前更新，导致判断 `localCurrentPage.value !== current` 恒为 false，事件被吞
**修复**：参考 [BaseTable.vue:554-568](file:///d:/workspace/lasudev/hyper-duty/frontend/src/components/BaseTable.vue#L554-L568)，把判断条件从 `localCurrentPage.value !== current` 改为 `props.pagination.currentPage !== current`

### 7.6 DBeaver 报 `syntax error at or near "ON"`

**症状**：执行 `INSERT ... ON CONFLICT DO NOTHING` 报语法错
**原因**：DBeaver 或部分代理工具不识别 `ON CONFLICT` 子句
**修复**：用 "DELETE + INSERT" 模式替代（见 6.1）
