⚠️ 存在的问题

数据模型

shadow_task_id 的设计问题

影子任务本身就是一个 pm_task，为什么需要单独的 shadow_task_id 字段？这增加了数据冗余。

当前: shadow_task_id 指向 pm_task.id
概念混淆

影子任务 vs 普通任务

影子任务本身也是一个 pm_task，但在 pm_task 表中没有明确标识它是影子任务。

问题: 需要查询 pm_task_shadow 才能知道是不是影子
功能缺失

缺少同步机制

源任务更新时，影子任务如何同步？当前设计没有处理。

问题: 源任务变了，影子不知道
交互体验

创建流程复杂

两步式创建虽然清晰，但对于快速创建影子任务可能过于繁琐。

问题: 需要选择任务 → 配置 → 创建
数据一致性

删除处理不完善

源任务被删除时，影子任务如何处理？当前没有级联处理逻辑。

问题: 源任务删了，影子变成孤儿

💡 改进方案

方案 A

简化数据模型（推荐）

核心思想

  • 在 pm_task 表添加 is_shadow 和 source_task_id 字段
  • 移除 pm_task_shadow 表，直接通过 pm_task 自身维护关系
  • 批注仍然放在 pm_shadow_annotation，关联到影子任务 id
erDiagram PM_TASK ||--o{ PM_TASK : "源任务 → 影子任务" PM_TASK ||--o{ PM_SHADOW_ANNOTATION : "影子任务有批注" PM_TASK { bigint id PK varchar task_name text description int status int progress boolean is_shadow bigint source_task_id FK text shadow_description } PM_SHADOW_ANNOTATION { bigint id PK bigint task_id FK text content varchar created_by datetime created_at }
✅ 优点
  • 查询简单，直接判断 is_shadow
  • 减少 JOIN，性能更好
  • 数据模型更直观
⚠️ 缺点
  • 需要修改现有 pm_task 表
  • 影子任务的元数据放在主表
方案 B

保留当前结构，优化体验

核心思想

  • 保持 pm_task_shadow 表结构不变
  • 添加同步机制：源任务更新时通知影子
  • 添加影子任务删除时的级联处理
  • 简化创建流程，支持快速创建
✅ 优点
  • 不需要大改现有代码
  • 渐进式优化风险小
  • 已有的功能都保留
⚠️ 缺点
  • 数据模型问题仍然存在
  • 查询复杂度没有改善

📝 总结与建议

我的推荐：渐进式优化

第一阶段：保持现有，优化体验

  1. 添加源任务删除时的级联处理逻辑
  2. 简化创建流程，提供快速创建模式
  3. 优化 UI 交互，让影子任务更直观

第二阶段：重构数据模型

  1. 在 pm_task 表添加 is_shadow 和 source_task_id 字段
  2. 迁移现有 pm_task_shadow 数据到 pm_task
  3. 简化查询逻辑，移除不必要的 JOIN

核心改进优先级

1 处理源任务删除的情况
2 优化影子任务标识的可见性
3 添加影子任务批量管理功能
4 考虑数据模型重构
