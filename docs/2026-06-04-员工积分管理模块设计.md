## 一、模块概述

员工积分管理模块用于对员工进行**事件驱动的积分评价**，结合**加班工时**进行综合评分，支持月度记录、季度/年度评选。

### 核心功能

| 功能 | 说明 |
|------|------|
| 积分事件管理 | 维护加分项/扣分项模板（如"团队赋能分享"、"抽检不合格"） |
| 积分记录录入 | 管理员为员工录入加减分记录 |
| 月度汇总 | 每月自动汇总积分 + 工时 → 综合评分 |
| 评选排名 | 按季度/年度汇总各月数据，生成排名 |

### 评分公式

```
月度综合评分 = 当月积分合计 × 积分权重(0.6) + 当月工时 × 工时权重(0.4)
```

- 积分合计：当月所有加分 + 扣分（扣分为负值）
- 当月工时：从 `duty_record` 表按员工+月份汇总 `overtime_hours`
- 权重可在周期配置中调整

### 3.3 工时数据来源

月度汇总生成时，从 `duty_record` 表按员工+月份汇总 `overtime_hours`：

```sql
SELECT employee_id, SUM(overtime_hours) AS work_hours
FROM duty_record
WHERE EXTRACT(YEAR FROM duty_date) = ? AND EXTRACT(MONTH FROM duty_date) = ?
GROUP BY employee_id
```

## 五、核心流程

### 5.1 积分录入流程

```
管理员通过 PersonSelector 选择员工 → 选择事件模板 → 确认分值 → 填写描述 → 保存到 score_record
```

### 5.2 月度汇总生成流程

```
1. 统计当月 score_record 按 employee_id 汇总 total_score
2. 查询 duty_record 按 employee_id 汇总 overtime_hours
3. 读取当前周期配置的点数权重和工时权重
4. 计算 comprehensive_score = total_score × 点权重 + work_hours × 工时权重
5. 写入 score_summary 表（存在则更新）
```

### 5.3 评选流程

```
选择周期类型（季度/年度）→ 选择年份和序号 → 
汇总 score_summary 中该期间内所有月份数据 → 
按 comprehensive_score 降序排名 → 展示
```

### 5.4 评选排名规则公示（2026-06-22 新增）

**背景**：评选排名是全员可见页面，但综合评分公式和权重对全员不透明，容易引发"暗箱"质疑。

**实现方案**：
- 在 `ScoreEvaluation.vue` 卡片标题"评选排名"后追加一个"?"图标（`el-icon` + `QuestionFilled`）
- 鼠标悬停触发 `el-popover`（placement=bottom-start，width=360px，trigger=hover）
- 悬浮提示内容分两部分：
  1. **计算规则**（固定文案）：
     ```
     月度综合评分 = 当月积分合计 × 积分权重 + 当月工时(h) × 工时权重
     评选排名 = 期间内各月综合评分的平均值
     ```
  2. **当前周期权重**（动态从后端读）：
     - 积分权重 / 工时权重：来自 `score_period_config` 中 `status=1` 的最新配置
     - 权重来源：`config`（管理员配置，黄色 tag）/ `default`（系统默认 0.6/0.4，灰色 tag）
     - 默认情况下显示一条提示："当前周期未配置权重，公示默认权重 0.6 / 0.4"

**新增接口**：

```java
// ScoreSummaryController.java
@GetMapping("/evaluation/config")
public ResponseResult<Map<String, Object>> getCurrentEvaluationConfig(
        @RequestParam String periodType,
        @RequestParam Integer year,
        @RequestParam Integer periodIndex) {
    Map<String, Object> config = scoreSummaryService.getCurrentEvaluationConfig(
        periodType, year, periodIndex);
    return ResponseResult.success(config);
}
```

**返回结构**：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "periodType": "QUARTERLY",
    "year": 2026,
    "periodIndex": 2,
    "pointWeight": 0.6,
    "hourWeight": 0.4,
    "source": "config"
  }
}
```

**前端调用**：在 `handleQuery`（查询排名按钮）里同步触发 `fetchEvalConfig`，保证切换周期后悬浮提示内的权重与当前查询周期一致；初次进入页面 onMounted 也调用一次。

**Service 实现要点**（`ScoreSummaryServiceImpl.getCurrentEvaluationConfig`）：

```java
// 查 score_period_config 中 status=1 的配置
ScorePeriodConfig config = scorePeriodConfigMapper.selectOne(
    new LambdaQueryWrapper<ScorePeriodConfig>()
        .eq(ScorePeriodConfig::getPeriodType, periodType)
        .eq(ScorePeriodConfig::getPeriodYear, year)
        .eq(ScorePeriodConfig::getPeriodIndex, periodIndex)
        .eq(ScorePeriodConfig::getStatus, 1)
);
if (config == null) {
    // 无配置时返回默认权重，前端用 source="default" 标识
    return defaultResult(periodType, year, periodIndex);
}
return configResult(periodType, year, periodIndex, config);
```

**测试用例**（10 条）：

| # | 输入 | 预期 |
|---|------|------|
| 1 | periodType=QUARTERLY, year=2026, periodIndex=2，有配置 | data.pointWeight=0.6, hourWeight=0.4, source=config |
| 2 | periodType=QUARTERLY, year=2026, periodIndex=2，无配置 | data.pointWeight=0.6, hourWeight=0.4, source=default |
| 3 | periodType=YEARLY, year=2025, periodIndex=1 | 返回年度配置或默认 |
| 4 | periodType=QUARTERLY, year=2099（未来） | 无配置走 default |
| 5 | periodType=quarterly（小写） | 后端严格校验，可能 400；前端已固定大写 |
| 6 | periodType 空 | @RequestParam 校验失败 400 |
| 7 | 前端 onMounted 默认 QUARTERLY + 当前年 + 当前季度 | 接口能正常返回 |
| 8 | 切换 periodType=YEARLY → periodIndex=1 | 接口仍能用，权重同步刷新 |
| 9 | 后端异常 | 前端 catch 兜底，evalConfig=null，悬浮提示仍可点击 |
| 10 | 多人同时查询 | 接口无状态、权重只读，支持并发 |

## 六、技术要点

- **后端**：Spring Boot 3 + MyBatis Plus，遵循现有分层架构
- **前端**：Vue 3 Composition API + Element Plus，员工选择统一使用 PersonSelector 通用选人组件
- **数据库**：PostgreSQL，SQL 脚本统一管理
- **工时对接**：只读查询 `duty_record` 表，不修改值班模块数据
- **无审批流程**：积分录入即生效，管理员直接操作
